## Outcome

✅ **Solved.**

Root cause: the frontend zod `ProposalStatus` enum in `risk-shared-components` was missing `superseded`. Because the list response is validated as `z.array(...)`, a **single** item with status `superseded` failed the whole-array validation → `APIError(422)` → with `throwOnError: false` the section silently returned `null` → comparisons "not appearing."

The agent then:

- Fixed the enum (+ `TradeComparisonStatus.ACCEPTED`, a latent second gap).
- Added the required `STATUS_VARIANT_MAP` entry and translations (TS would have broken otherwise).
- Extended the feature (list shows status; action gated to `PENDING` only).
- Added a focused test for the new behavior.
- Propagated the fix to the second MFE (`ops-web-risk`), catching that it uses an **injected** pnpm `file:` dep (stale copy, needed reinstall) vs ebury's live `link:`.
- Updated the CLT-348 handoff summary, including correcting its own earlier claims.

## What StackPilot did well

1. **Empirical verification over speculation (eventually).** The decisive move was reproducing the exact API payload from the live DB and round-tripping it through the _built_ zod schema (a throwaway vitest spec). That converted a stack of hypotheses into the actual bug (`received: "superseded"`). It used the running backend (`:8002`), real Postgres (`:5433`), and Python in the service venv to serialize a `TradeComparisonListItem` exactly as the endpoint would.

2. **Self-correction under uncertainty.** It floated ~5 competing hypotheses (stale dist, broken barrel export, 403 auth gate, camelCase mismatch, feature flag) and killed each with evidence rather than committing. It explicitly caught its own false positive: after editing the `ACCEPTED` enum it discovered the dist already contained it and openly walked the reasoning back.

3. **Clean diagnostic hygiene.** Temporary colored-box diagnostics were injected into the render tree to get runtime signal without a debugger, then **all** reverted and verified clean via `git diff --stat`.

4. **Full blast-radius thoroughness.** Beyond the fix: variant-map + translation updates, the injected-vs-linked pnpm distinction between the two MFEs, the fact that the two MFEs gate on **different** flag keys (`risk.hedge.multi_ti_proposal` vs `risk.ops_web_risk.multi_ti_proposal`), and an accurate handoff addendum that corrected its own earlier statements (dist is gitignored; lockfile churn from the reinstall).

5. **Appropriate restraint.** Asked before killing the user's running dev server; left crowdin translation sync to the normal pipeline instead of guessing.

## What it did less well

1. **~30 steps of static-analysis theorizing before getting empirical.** Steps 5–29 chased the export/barrel-resolution hypothesis through repeated greps — and grep on `export *` barrels _structurally cannot_ reveal re-exported names, which the agent only realized at step 27. The winning technique (validate the real payload against the schema) was available from the start; a reproduction-first instinct would have cut the session substantially.

2. **A speculative edit before evidence.** It edited the `ACCEPTED` enum before confirming it was the problem, then found the change redundant. Harmless here, but it's editing-before-evidence.

3. **Real cost waste flagged by the profiler.** One cold-cache regeneration re-billed **108,895 tokens** after a ~10 min idle gap (the single largest waste across both sessions). Long stretches of sequential single-file greps grew context from ~2 → ~157k tokens with many low-value round-trips.

4. **Zero parallelism (structural).** The session ran entirely one-tool-call-at-a-time. It won on speed/cost here through targeted calls once it found its footing, but on a broader multi-concern task the lack of fan-out is a ceiling.

## Comparison with Claude Code

The most instructive difference is orchestration:

- **Claude Code fanned out** — dispatched parallel exploration sub-agents across the three frontends and batched independent reads/greps ("investigate the three concerns in parallel"), and front-loaded confirming each stated requirement.
- **StackPilot ran fully sequentially** — yet finished faster and cheaper on this narrower task due to fewer, more targeted calls.

Claude Code's session carried its own inefficiencies: the profiler flagged **14 ordered search→read chains** and a repeated-8× identical Bash call. Neither session is a clean efficiency winner across the board.

## Bottom line

StackPilot handled a genuinely hard, under-specified, multi-repo bug **correctly and thoroughly**, with excellent self-correction and cleanup — the class of task where many agents ship a plausible-but-wrong fix (the tempting `ACCEPTED` enum edit) and stop.

Its principal weakness is **sequencing discipline**: too much static grep-theorizing before reaching for empirical verification, plus no parallelism. Leading with "reproduce the payload against the schema" and batching independent lookups could have reached the same result in roughly half the steps.

### Recommendations

1. **Reproduction-first bias:** when a symptom is "data not appearing" and a live backend/DB is available, validate the real payload against the consuming schema _before_ extended static analysis.
2. **Evidence before edits:** avoid speculative source edits until a hypothesis is confirmed.
3. **Parallel exploration:** batch independent reads/greps and consider sub-agent fan-out for multi-file / multi-repo investigation.
4. **Cache-cost awareness:** the 108k-token cold-cache regeneration after idle is a recurring Bedrock cost pattern worth surfacing to the user or mitigating.
